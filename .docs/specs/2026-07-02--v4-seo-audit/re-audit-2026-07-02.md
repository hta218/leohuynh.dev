## Finding 1 — `seo:check` does not match rendered gist titles

Severity: Medium

### Issue

`scripts/check-seo-lengths.mjs` now checks snippet `heading` before `title`:

- `seoTitle ?? heading ?? title`

But `src/pages/gists/[...slug].astro` renders metadata title as:

- `seoTitle ?? title`

This means `bun run seo:check` underreports snippet title length issues. It says those gist titles are short because `heading` is short, while the actual rendered `<title>` uses the longer `title` frontmatter field.

### Evidence

Files:

- `scripts/check-seo-lengths.mjs`
- `src/pages/gists/[...slug].astro`
- `data/snippets/*.mdx`
- rendered `dist/client/gists/*/index.html`

Examples where the script's effective title is short but rendered title is long:

| File | Script checks | Actual rendered title | Actual len |
| --- | --- | --- | ---: |
| `data/snippets/connecting-to-github-using-ssh.mdx` | `Connecting to GitHub using SSH — Leo Huynh` | `Enhance Github security by connecting using Secure Shell Protocol (SSH) — Leo Huynh` | 83 |
| `data/snippets/find-and-kill-process-on-given-port-in-macos.mdx` | `Kill that process — Leo Huynh` | `Find which process is running on a certain port and kill it — Leo Huynh` | 71 |
| `data/snippets/parse-function-from-string.mdx` | `Parse functions from string — Leo Huynh` | `How (and why) to parse a function from a string in JavaScript/TypeScript — Leo Huynh` | 84 |
| `data/snippets/using-spotify-api-to-display-currently-playing-track.mdx` | `Display Spotify currently playing track — Leo Huynh` | `How to work with Spotify's API to display currently playing track on a website? — Leo Huynh` | 91 |

Full mismatch scan:

- Published snippet metadata mismatches between script and rendered output: `23`
- Actual long rendered gist titles missed by script: `14`

### Impact

The SEO check becomes less trustworthy. Future agents may think snippet SEO titles are clean because the script checks `heading`, while Google sees the longer rendered `title`.

### Recommended fix

Align the script with the route, or align the route with the script. Pick one policy and make both surfaces match.

Recommended low-risk fix:

1. Keep visible gist H1 as `heading`.
2. Keep metadata title as `seoTitle ?? title` in `src/pages/gists/[...slug].astro`.
3. Update `scripts/check-seo-lengths.mjs` so snippets use `seoTitle ?? title`, not `heading`.
4. Add a short comment explaining that `heading` is display-only for snippets.
5. Re-run `bun run seo:check` and expect the warning count to increase unless `seoTitle` fields are added to long snippets.

Alternative:

- Change `src/pages/gists/[...slug].astro` to `seoTitle ?? heading` for metadata. This would shorten rendered gist titles, but it changes public SEO titles for existing pages. Do this only if the owner accepts that change.

### Acceptance criteria

- `scripts/check-seo-lengths.mjs` computes the same effective title as the rendered route for blog, gist, and misc pages.
- A mismatch scan between script-effective titles and rendered `<title>` returns `0` unexpected mismatches.
- `bun run seo:check` output is documented as informational and accurately reflects rendered metadata.
- `bun run check` and `bun run build` pass.

## Finding 2 — `/dotfiles` title duplicates the section name

Severity: Low

### Issue

`/dotfiles` is indexable and included in the sitemap, but its document title is:

- `dotfiles — dotfiles — Leo Huynh`

This happens because `src/pages/dotfiles/[...slug].astro` passes:

- `title={`${title} — dotfiles`}`

For the index route, `title` is already `dotfiles`, so the section suffix repeats.

### Evidence

`vercel dev` check for `/dotfiles`:

- status: `200`
- canonical: `https://www.leohuynh.dev/dotfiles`
- robots: none (indexable)
- title: `dotfiles — dotfiles — Leo Huynh`
- Lighthouse SEO score: `100`

Source:

- `src/pages/dotfiles/[...slug].astro:29-37`

### Impact

Low SEO impact because Lighthouse still passes and the title is short, but the title looks sloppy in browser tabs/search snippets and weakens polish for a sitemap-submitted page.

### Recommended fix

Special-case the index title:

- index route: pass `title="dotfiles"`
- nested dotfile route: pass a composed title like `<slug> — dotfiles`

Example:

```ts
const pageTitle = isIndex ? 'dotfiles' : `${title} — dotfiles`
```

```astro
<BaseLayout title={pageTitle} ...>
```

### Acceptance criteria

- `/dotfiles` renders `<title>dotfiles — Leo Huynh</title>`.
- Nested dotfile pages still render useful titles like `<path> — dotfiles — Leo Huynh`.
- `/dotfiles` remains canonical, indexable, and in the sitemap.
- `bun run check` and `bun run build` pass.
