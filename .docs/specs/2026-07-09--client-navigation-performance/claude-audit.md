## Overall verdict

The spec's diagnosis (SSR/data blockers vs. static-payload blockers, route classification, P0→P1→P2 ordering) is directionally sound and matches the actual source. However, I found **two blocking issues** that must be corrected before implementation begins — both are places where the plan's proposed *solution* (not the diagnosis) could introduce a real regression or a moderation-bypass data leak. Everything else is a documentation-accuracy or clarity gap.

## Blocking issues

1. **P1.1 cache proposal risks leaking unapproved/pending guestbook entries via CDN cache.**
   `src/pages/api/guestbook/index.ts:16-19,36-56` (existing `GET /api/guestbook`) currently returns `cache-control: no-store` precisely because the response varies per request on the session cookie: `isAdmin` controls whether the SQL filter includes non-`'approved'` rows (`src/lib/guestbook/queries.ts:44`), and `toEntryDTO()` includes a `status` field on every entry (`src/lib/guestbook/mappers.ts:21-40`). plan.md:117-118 proposes adding `public, s-maxage=30-60` "for anonymous approved entries if safe" without specifying that this must be a *separate response branch keyed off `isAdmin`/viewer*, and Vercel's edge cache does not vary by cookie by default. If implemented naively (same endpoint, same cache-control regardless of viewer), an admin's response — including pending/rejected entries and their `status` — could be cached and served to anonymous users, or an anonymous 30-60s-stale cache could mask an admin's own just-submitted moderation action. This is a concrete moderation-bypass/data-leak risk, not a hypothetical.
   → Must be corrected in the spec before P1.1 is implemented: cache-control on this route must only ever be `public, s-maxage=...` on the exact code path where `viewer` is absent/anonymous and `isAdmin` is `false`; every other path stays `no-store`.

2. **Viewport prefetch on a persistent, always-visible sidebar behaves like eager/on-every-load prefetch, not "prefetch when scrolled into view."**
   Confirmed in `node_modules/astro/dist/prefetch/index.js:80-89`: the `viewport` strategy attaches an `IntersectionObserver` to matching anchors on `onPageLoad`. Since the StudioShell left explorer is persistent and normally fully visible without scrolling, `IntersectionObserver` reports these links as intersecting immediately on essentially every page load/document swap across the site — not occasionally. plan.md:76 explicitly leaves the door open to using `viewport` prefetch on `/guestbook` ("if backend load is low... otherwise use hover/tap"), but frames it as a measurement question rather than stating the real mechanism: `viewport` on `/guestbook` would fire a Supabase-backed fetch on nearly every page view sitewide, not just when a user actually navigates there. Given findings.md's own measured 1.3-2s TTFB on that route, this is a real backend-load/cost risk, not a minor tuning knob.
   → Must be corrected before P0.2 is implemented: `/guestbook` should use `hover` or `tap` only, never `viewport`, as a hard rule (not a "measure and decide" option) until P1.1 removes the query from the request path.

## Non-blocking improvements

- **Astro version imprecision**: findings.md:24 says "Astro 6/7"; `package.json` pins `^7.0.6` and the installed version is exactly `7.0.6`. Minor wording fix.
- **Dotfiles memoization framing is misleading**: `src/lib/dotfiles.ts` memoizes tree/entries in a module-level `Map` (TTL 1h), but this only helps *warm* serverless instance reuse — it does nothing for the cold-start case that findings.md itself identifies as the actual root cause. plan.md's P1.2 "Current problem" section should say this explicitly so the Option A (static snapshot) recommendation reads as necessary rather than optional.
- **Unmeasured one-hop routes**: `src/pages/blog/page/[page].astro` (blog pagination) and `src/pages/topics/[tag].astro` (tag pages) are reachable one click from sidebar routes and were not measured or mentioned, even though the audit scope is "data-heavy routes reachable from the left navigation." Worth at least a footnote acknowledging they're out of scope for this pass, or a quick size/TTFB spot check, especially if any tag has a large post count.
- **Vercel "cache-control normalization" claim is unverifiable and slightly overstated**: `src/pages/dotfiles/[...slug].astro:21-24` does set the full `public, s-maxage=3600, stale-while-revalidate=86400` string in source — the header is not silently dropped by anything in this repo (`vercel.json` and the `@astrojs/vercel` adapter middleware were checked, neither touches `cache-control`). The observed truncation to bare `public` is plausibly a platform/edge-layer behavior, but findings.md states it as established fact. Should be softened to "observed, cause not confirmed in this repo."
- **`astro:before-preparation` claim — I verified this one directly and it's actually correct, flagging only so it isn't second-guessed later**: it's dispatched at runtime (`node_modules/astro/dist/transitions/events.js:35-40,99`) even though it doesn't appear in the narrower `Events` type union in `events.d.ts:2` (that union only covers the plain-`Event` cases; the richer `TransitionBeforePreparationEvent`/`TransitionBeforeSwapEvent` classes are typed separately). No spec change needed here, but implementers may need an ambient type cast (`as EventListener` or similar) since it may not autocomplete.

## Recommended spec edits

- plan.md P1.1 step 3: replace "existing or new: `GET /api/guestbook/entries?cursor=...`" with a reference to the endpoint that already exists — `GET /api/guestbook` (`src/pages/api/guestbook/index.ts:36-56`) — which already derives `viewer`/`isAdmin` from the session cookie and reuses the same safe `getGuestbookEntries()` filtering. The task is "wire `GuestbookWall` to consume this existing route," not "build a new one."
- plan.md P1.1 step 6: make explicit that any added `s-maxage` caching must branch on `viewer`/`isAdmin` being absent/false, and must never share a cache entry between anonymous and authenticated/admin responses (see Blocking issue #1).
- plan.md P0.2: change the `/guestbook` prefetch guidance from "viewport acceptable if backend load is low" to a hard rule: `hover`/`tap` only, revisit after P1.1 lands (see Blocking issue #2). Apply the same caution to `/llms` (lower severity since it's CDN-cached, but cache-miss windows would still fan out the GitHub-token-gated fetch on every page load).
- findings.md: soften the Vercel cache-control-normalization statement to "observed behavior, root cause not confirmed in this repository."
- findings.md/plan.md P1.2: add a line clarifying that in-memory memoization only helps warm-instance reuse, not the cold-start case being fixed.

## Implementation cautions

- Before touching P0.2, confirm empirically (via the sidebar's actual scroll container) whether all nav links are truly always in-viewport with no scroll — if so, treat `viewport` prefetch as equivalent to `load` prefetch for every route it's applied to, and budget backend/CDN load accordingly even for the static-route recommendations (`/shelf`, `/heatmap`, `/builds`, `/topics`, `/uses`).
- When implementing P1.1, write (or extend) a test/manual check that an anonymous request never receives a cached response that was generated for `isAdmin: true`, and vice versa — this is the one place in this plan where a caching decision is also an auth-boundary decision.
- P0→P1→P2 ordering itself is sound (low-risk UX/prefetch → real server blockers → large static payload refactors, with a re-measure gate before P2). The only sequencing change needed is the one above: don't let P0.2's prefetch strategy for `/guestbook`/`/llms` ship ahead of/independent from the P1 fixes for those same routes, since prefetching a still-slow backend on every page load could make things measurably worse before P1 lands.
